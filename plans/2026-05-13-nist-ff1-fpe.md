# Block 5 — NIST FF1 FPE + rename existing strategy

> **Status:** plan — part of the 2026-05-13 audit-fix series.
> **Standalone:** no dependency on Blocks 1–4.
> **Repos touched:** `decoy-engine` (primary), `decoy-platform` (light UI label tweak).

## Why

`transforms/fpe.py:14-21` candidly admits:

> Design note: this is not NIST FF1 (which requires AES-CBC and therefore the `cryptography` package). The Feistel+HMAC approach has the same user-visible properties — format-preserving, bijective, keyed-deterministic — using the HMAC-SHA256 primitive that is already in every other keyed transform. Defer a hard AES dep until a customer asks for NIST SP 800-38G compliance by name.

The reference (§1, §6) flags this as a defensibility risk: NIST SP 800-38Gr1 (2nd public draft, 2025) **withdraws FF3/FF3-1, leaving FF1 as the only NIST-approved FPE mode**. The current strategy is bijective and keyed but won't pass an auditor's "is this NIST-approved?" question.

This block:
1. Renames the existing strategy to `format_preserving_feistel` to be honest about its construction.
2. Adds a true NIST FF1 strategy at `fpe_ff1` for customers who need 800-38G compliance.
3. Keeps the old `fpe` key as a deprecation alias for one release.
4. Enforces NIST's minimum domain size 1,000,000 on the new path.

## Engine changes

### 2. New module: `transforms/fpe_ff1.py`

Key choices:
- **Algorithm:** NIST SP 800-38G FF1 with AES-128 (AES-256 optional, defaults to 128 because PAN-shaped inputs don't justify the wider key).
- **Library:** `cryptography.hazmat.primitives.ciphers.algorithms.AES` + `modes.ECB`. PyCA `cryptography` is the right choice; the alternative `pyffx` is an FF1 toy implementation and not audited.
- **Minimum domain size:** `domain = radix ** length`. Reject `domain < 1_000_000` with `FPEDomainTooSmall`.
- **Tweak:** the column name (UTF-8). FF1 takes an arbitrary tweak; this gives per-column domain separation.

Pseudocode skeleton:

```python
# transforms/fpe_ff1.py
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.backends import default_backend
from decoy_engine.transforms.base import BaseMaskingStrategy

_MIN_DOMAIN = 1_000_000  # NIST SP 800-38G Rev 1


class FF1Strategy(BaseMaskingStrategy):
    """NIST SP 800-38G FF1 format-preserving encryption."""

    def apply(self, column, rule, *, context_df=None):
        charset = self._resolve_charset(rule.get("charset", "digits"))
        radix = len(charset)
        key = self._column_key(rule.get("column"))
        if key is None:
            raise ConfigError("fpe_ff1 requires a master key (no legacy fallback)")
        tweak = rule.get("column", "").encode("utf-8")
        validate_luhn = bool(rule.get("validate_luhn", False))

        # one min-domain check per column, on the first non-null value's length
        first = column.dropna().astype(str).head(1)
        if len(first):
            length = len(first.iloc[0])
            if radix ** length < _MIN_DOMAIN:
                raise FPEDomainTooSmall(
                    f"radix^length = {radix}^{length} = {radix**length} "
                    f"< NIST minimum {_MIN_DOMAIN}; increase length or charset"
                )
        return column.map(
            lambda v: self._ff1_encrypt(str(v), key[:16], tweak, charset, radix, validate_luhn)
            if not pd.isna(v) else v
        )

    def _ff1_encrypt(self, plain, key, tweak, charset, radix, validate_luhn):
        # standard FF1 round-function implementation; ~150 lines.
        # Reference: NIST SP 800-38G Algorithm 7.
        ...
```

The FF1 round function is the canonical PRF over `AES_CBC_MAC(key, …)`; pseudocode in NIST SP 800-38G §6.2 Algorithm 7. Implement carefully and test against the NIST test vectors. Don't write a custom variant.

### 3. Refuse the legacy path on `fpe_ff1`

FF1 is only honest when the master key is configured; the seeded-SHA256 fallback that `format_preserving_feistel` permits has no FF1 analog. Raise on missing master key. Customers who want a key-less FPE have `format_preserving_feistel`.

## Risks

- **FF1 implementation correctness.** This is the single highest-risk part of the block. Use a vetted reference (e.g. mirror Botan's or OpenFPE's FF1 closely) and lean on NIST KATs. Get a second pair of eyes before merging.
- **`cryptography` dep:** already in the platform repo for Fernet; pinning in engine is a small bump.
- **Marketing copy:** anywhere the website or docs say "FPE" today needs to say either "FPE (FF1)" or "format-preserving Feistel" depending on configuration. Out of scope for this block; track in a follow-up content task.
