## 1. Summary

The engine's determinism core (`determinism/`, `internal/crypto.py`, `_hkdf.py`) is
well-designed and the recent QA-10 / QA-7 / QA-1 series of fixes is substantive. The
most critical gap is a **V1/V2 parity break in `generation/synthesize._reference`**: V1
`_generate_reference_column` was updated to sort the FK pool (QA walks/generators F1)
while the V2 counterpart still uses insertion-order unique — meaning the two generators
produce different FK assignments for the same seed on any source with non-sorted parent
values. The next most impactful cluster is that both `HashStrategyHandler` and
`DateShiftStrategyHandler` call `derive()` per row, ignoring the `DeriveContext`
pre-computation class that was introduced precisely to eliminate this overhead — costing
~25 s per 1M-row column. Additionally, `DateShiftStrategyHandler` will raise at runtime
when the source column has a tz-naive `datetime64[ns]` dtype, and
`_apply_cardinality_bounds` has an O(n × |ref_values|) loop that becomes O(n²) for
wide FK pools.

### F-1 — CRITICAL | Correctness / Determinism
**`generation/synthesize.py:457-464` — `_reference` uses insertion-order unique; V1 was updated to sort (F1 parity break)**

V1 `_generate_reference_column` (`generators/columns.py:942-946`) was updated by the
QA walks/generators F1 fix (2026-06-01) to sort the FK pool:

```python
# columns.py (V1) — F1 fix
raw_pool = ref_df[reference_column].dropna().unique().tolist()
try:
    ref_values = sorted(raw_pool)
except TypeError:
    ref_values = sorted(raw_pool, key=str)
```

V2 `synthesize._reference` still uses insertion-order unique:

```python
# synthesize.py (V2) — NOT updated
seen: set = set()
ref_vals: list[Any] = []
for v in raw_vals:
    if v is None or v in seen:
        continue
    seen.add(v)
    ref_vals.append(v)
```

The comment on this block says it mirrors "V1 `dropna().unique()` on a pandas Series",
but V1 was explicitly changed away from that pattern to sort. Since `ref_rng` is seeded
deterministically from `col_seed`, a different pool order produces different FK
assignments for the same seed. This is a byte-output break between V1 and V2 for any
reference column whose parent values arrive in non-sorted order (the common case for
DB-sourced data).

**Fix:** Mirror the V1 F1 fix in `_reference`:

```python
raw_vals = parent_tbl.column(ref_column).to_pylist()
# Drop None, then sort (mirrors V1 F1 fix in columns.py:942-946).
non_null = [v for v in raw_vals if v is not None]
try:
    ref_vals = sorted(set(non_null))
except TypeError:
    ref_vals = sorted(set(non_null), key=str)
if not ref_vals:
    return [None] * n
```

Note: `sorted(set(...))` produces the same sorted-unique sequence as V1's
`sorted(raw_pool)` where `raw_pool = dropna().unique().tolist()`.

### F-2 — HIGH | Correctness
**`execution/_strategies/_date_shift.py:63` — `_canonicalize_source` raises for tz-naive `Timestamp` columns**

`DateShiftStrategyHandler.run` applies `col.astype(object)` only when the column dtype
is an extension array (line 51-52). Standard `datetime64[ns]` is NOT an extension array
dtype, so it is NOT cast to object, and `for i, value in enumerate(col)` yields tz-naive
`pandas.Timestamp` objects. `_canonicalize_source` detects `isinstance(value, datetime)`
with `value.tzinfo is None` and raises `GenerationError(code='timezone_naive_datetime')`.

**Reproduction:**
```python
import pandas as pd
from decoy_engine.generation.pool._canonicalize import _canonicalize_source
ts = pd.Timestamp("2023-01-15")   # tz-naive
_canonicalize_source(ts)           # raises GenerationError
```

The strategy is most commonly applied to date-string columns (where `value` is a Python
`str` and the str-path is taken). But any operator using `date_shift` on an actual
`datetime64[ns]` column will crash.

**Fix:** Widen the astype guard to cover standard datetime columns too, converting
`Timestamp` to its string representation under the detected format before canonicalizing:

```python
# At the top of the shift loop, derive canonical bytes from the
# formatted string (not the raw cell) so tz-naive Timestamps don't
# reach _canonicalize_source.  The formatted string is stable for the
# same source date, so "same date -> same shift" is preserved.
for i, value in enumerate(col):
    if unusable[i]:
        shifts.append(0)
        continue
    # Use the string representation of the parsed date so tz-naive
    # Timestamps never reach _canonicalize_source.
    src_str = parsed.iloc[i].isoformat() if not isinstance(value, str) else str(value)
    digest = derive(ctx.job_seed, plan.namespace, src_str.encode("utf-8"))
    shifts.append(min_days + (int.from_bytes(digest[:8], "big") % range_size))
```

Alternatively, extend the existing extension-array guard:
```python
if pd.api.types.is_extension_array_dtype(col.dtype) or pd.api.types.is_datetime64_dtype(col.dtype):
    col = col.astype(str)
```
and remove the `_canonicalize_source` call entirely (the string path in
`_canonicalize_source` applies NFC normalization, which is a no-op for ISO date strings).

### F-3 — HIGH | Performance
**`execution/_strategies/_hash.py:52-56` and `_date_shift.py:59-64` — `derive()` called per row, ignoring `DeriveContext`**

`DeriveContext` was introduced (QA-10 F4, 2026-06-01) precisely to amortize the HKDF
key computation: at 1M rows/column, calling `derive(seed, namespace, source)` per row
wastes 2x HMAC-SHA256 per row (the key derivation step). The docstring on `DeriveContext`
estimates ~25 s/column on a modern core for a 50-column table.

`HashStrategyHandler.run` (`_hash.py:52-56`) does:
```python
for i, value in enumerate(source):
    if na_mask[i]:
        ...
    token = derive(ctx.job_seed, plan.namespace, _canonicalize_source(value)).hex()
```

`DateShiftStrategyHandler.run` (`_date_shift.py:59-64`) does:
```python
for i, value in enumerate(col):
    ...
    digest = derive(ctx.job_seed, plan.namespace, _canonicalize_source(value))
```

Both recompute `hkdf_sha256(seed, salt, namespace)` once per row when it is constant
for the entire column.

**Fix (hash strategy):**
```python
from decoy_engine.determinism._derive import DeriveContext

ctx_d = DeriveContext.for_column(ctx.job_seed, plan.namespace)
for i, value in enumerate(source):
    if na_mask[i]:
        out.append(None)
        continue
    token = ctx_d.derive_source(plan.namespace, _canonicalize_source(value)).hex()
    out.append(token[:truncate] if truncate is not None else token)
```

Apply the same pattern to `DateShiftStrategyHandler.run`.

**Profile gate:** `python -m timeit` with a 100K-row column will show ~2x speedup for
`hash` and `date_shift` strategies purely from removing redundant HKDF work.

### F-4 — HIGH | Performance
**`generators/columns.py:1048-1075` — `_apply_cardinality_bounds` is O(n × |ref_values|)**

Two inner list-comprehensions scan the entire `values` list (length n) once per element
in `ref_values`:

```python
# Step 1 (line 1050-1053):
for pv in ref_values:
    if counts[pv] > max_eff:
        indices = [i for i, v in enumerate(values) if v == pv]  # O(n) per pv

# Step 3 (line 1071-1073):
for pv in ref_values:
    candidates = [i for i, v in enumerate(values) if v == pv and i not in already_free]  # O(n) per pv
```

For a table with 1M rows and a FK pool of 1 000 distinct parent values, this is 2 × 10⁹
comparisons. Building a `{value: [indices]}` index once reduces both to O(n) total.

**Fix:**
```python
# Build index once before steps 1-3.
from collections import defaultdict
idx_by_value: dict[Any, list[int]] = defaultdict(list)
for i, v in enumerate(values):
    idx_by_value[v].append(i)

# Step 1: use idx_by_value[pv] directly.
for pv in ref_values:
    if counts[pv] > max_eff:
        excess = counts[pv] - max_eff
        indices = list(idx_by_value[pv])
        rng.shuffle(indices)
        free_slots.extend(indices[:excess])
        # Update the index to reflect freed slots.
        for i in indices[:excess]:
            idx_by_value[pv].remove(i)  # or rebuild after this loop
        counts[pv] = max_eff
```

### F-5 — HIGH | Correctness / Determinism
**`generation/pool/_canonicalize.py:78-85` — `np.bool_` takes `numbers.Integral` path; Python `bool` takes dedicated path — inconsistent encoding for boolean columns**

The dispatch is:
```python
if isinstance(value, bool):          # True  -> b"\x01"
    return b"\x01" if value else b"\x00"
if isinstance(value, numbers.Integral):
    return _encode_int(int(value))   # np.bool_(True) -> b'\x00\x00\x00\x01\x01'
```

`isinstance(True, bool)` is `True` (Python `bool` is a subclass of `int`), so Python
booleans correctly take the first branch. `isinstance(np.bool_(True), bool)` is `False`,
but `np.bool_` is registered with `numbers.Integral` (numpy registers its integer types
with the ABC), so `np.bool_(True)` takes the second branch and produces a different
encoding:

- Python `True`  → `b"\x01"` (1 byte)
- `np.bool_(True)` → `b'\x00\x00\x00\x01\x01'` (5 bytes: 4-byte length prefix + 1-byte body)

When pandas returns `np.bool_` scalars from a boolean dtype Series (which it does for
numpy-backed boolean arrays, `Series.iloc[i]`), the canonical bytes differ from what a
test that passes Python `True` would produce, breaking cross-run stability if the column
dtype changes between production and test environments.

**Fix:** Add an `np.bool_`-specific guard before the `numbers.Integral` branch:

```python
import numpy as np

if isinstance(value, bool) or type(value).__name__ == "bool_":
    # Covers Python bool AND numpy bool_ scalar.
    return b"\x01" if value else b"\x00"
```

Or more robustly:
```python
if isinstance(value, (bool, np.bool_)):
    return b"\x01" if value else b"\x00"
```

This requires importing numpy, which `_canonicalize.py` already implicitly depends on
via the `numbers.Integral` path; making the numpy import explicit is cleaner.

**Verify:** `assert _canonicalize_source(True) == _canonicalize_source(np.bool_(True))`
should pass after the fix.

### F-6 — MEDIUM | Performance
**`generation/pool/_sampler.py:241-246` — `_match_source_cardinality` uses `.iloc[i]` per row (same anti-pattern fixed in `_deterministic` by Q6)**

The S21 Q6 fix pre-materialized source + null mask to lists in `_deterministic` to avoid
O(n) pandas scalar-unboxing overhead. `_match_source_cardinality` was not updated:

```python
is_null = source.isna()          # pd.Series
for i in range(n):
    if is_null.iloc[i]:          # O(1) but repeated n times with pandas overhead
        output.append(pd.NA)
    else:
        output.append(value_map[source.iloc[i]])  # same problem
```

**Fix:** Materialize before the loop, mirroring `_deterministic`:
```python

src_values = source.tolist()

    if is_null_arr[i]:
        output.append(pd.NA)
    else:
        output.append(value_map[src_values[i]])
return pd.Series(output)
```

### F-8 — MEDIUM | Correctness / Data
**`generators/columns.py:867` — nanosecond offset arithmetic loses float64 precision for time spans > ~4 months**

`_generate_distribution_datetime` computes per-row timestamp offsets as:

```python
rand_floats = rng.random(size=num_rows)             # float64 in [0,1)
offsets = (rand_floats * (hi_ns - lo_ns)).astype(np.int64)
```

`hi_ns - lo_ns` is a nanosecond span. Float64 has 52 bits of mantissa (~9.0 × 10¹⁵).
For spans > ~9 × 10¹⁵ ns (≈ 104 days), the product rounds to the nearest representable
float, introducing visible time quantization:

| Span | Quantization error |
|------|--------------------|
| 1 year | ~35 ns |
| 10 years | ~350 ns |

For practical date ranges (snapshot min-max spanning years), the quantization is well
below microsecond resolution — not visible in any real-world datetime field — but the
arithmetic could be made exact at no cost:

**Fix:** Use `rng.integers` on scaled values or compute via two integer draws:
```python
# Integer-safe per-row sampling for arbitrary spans.
span = (hi_ns - lo_ns).astype(np.int64)          # may be large but exact
# Decompose span into coarse (ms) + fine (ns within ms) to stay in int64.
ms_span = span // 1_000_000
fine_span = span % 1_000_000
coarse_ns = rng.integers(0, np.maximum(ms_span, 1), size=num_rows) * 1_000_000
fine_ns = rng.integers(0, 1_000_000, size=num_rows) % np.maximum(fine_span, 1)
offsets = (coarse_ns + fine_ns).clip(0, span - 1)
```

Or more simply, accept the ~35 ns quantization as documented with a short comment.

### F-9 — MEDIUM | Correctness / Determinism
**`generators/columns.py:979` — `weights` read inside loop for `weighted` distribution**

In `_generate_reference_column`, the per-row loop for `distribution == "weighted"`:

```python
for i in range(num_rows):
    ...
    elif distribution == "weighted":
        weights = column_config.get("weights")      # re-read every row
        if not weights or len(weights) != len(ref_values):
            weights = None
        values.append(ref_rng.choices(ref_values, weights=weights, k=1)[0])
```

`column_config.get("weights")` is O(1) but the condition `len(weights) != len(ref_values)`
is re-evaluated on every row. If `ref_values` grows across iterations (it doesn't — it's
pre-built above the loop), this could silently change. More importantly, the length check
`len(weights) != len(ref_values)` uses the SORTED pool size, but `weights` in the YAML
was designed to map to the original insertion-order values. After the F-1 fix lands
(sort the pool), a YAML `weights` list that was authored against unsorted value order
will silently misalign with the sorted pool and produce wrong relative probabilities.
This is a latent bug activated by the F-1 sort fix.

**Recommended fix:** Hoist before the loop and document that `weights` must align with
sorted `ref_values` (or re-sort them together).

### F-10 — LOW | Reliability
**`internal/crypto.py:64` — `hmac_seed` returns `0` for `None` input (all nulls share seed 0)**

```python
def hmac_seed(key: bytes, value: Any) -> int:
    if value is None:
        return 0
```

Any strategy or provider that calls `hmac_seed` to seed a Faker instance for a nullable
column will generate identical Faker output for every null row (they all seed with 0),
then when the null rows are restored, those identical values are dropped. This is benign
for masking (nulls are preserved elsewhere). However, if a caller mistakenly omits the
null guard before calling `hmac_seed` and inserts the result into the output, every null
row maps to the same non-null Faker value — a silent data-level inconsistency. No
current caller in the engine hits this, but the contract is fragile.

**Recommended fix:** Return `None` when `value is None`, and require callers to handle
it explicitly. Or raise `ValueError("hmac_seed: value must not be None; null guard upstream")`.

### F-12 — NIT | Correctness
**`generators/columns.py:50` — HMAC block-size comment is wrong (32 bytes vs 64 bytes)**

```python
# Repeated 4x to fill a 32-byte buffer (HMAC-SHA256's natural block size);
seed_bytes = (local_seed & ((1 << 64) - 1)).to_bytes(8, "big")
return hmac_hex(seed_bytes * 4, value)[:8]
```

HMAC-SHA256's internal block size (B in RFC 2104) is 64 bytes (SHA-256 operates on
512-bit = 64-byte blocks). The key produced here is 32 bytes, which HMAC-SHA256 handles
correctly (keys shorter than B are zero-padded internally). The code is correct; the
comment is wrong. Change "32-byte buffer" to "key length >= 16 bytes" or simply remove
the parenthetical.

### F-13 — NIT | Design
**`execution/_strategies/_faker.py:73-74` — per-row list assignment can be replaced with vectorised pd.Series**

```python
values = list(sampled)
df[column] = [None if na_mask[i] else values[i] for i in range(n)]
```

`sampled` is already a `pd.Series`. The null-restore can be done with pandas:

```python
sampled_series = pd.Series(sampled.to_numpy(), dtype=object)
sampled_series[na_mask] = None
df[column] = sampled_series
```

This avoids the Python-level list comprehension over n elements. The speedup is
proportional to n (C-level boolean indexing vs Python loop).

## 3. Performance Notes

**Bottleneck classification (by strategy):**

| Strategy | Primary bottleneck | Secondary |
|----------|--------------------|-----------|
| `hash` | CPU: HMAC-SHA256 per row + redundant HKDF (F-3) | Python loop overhead |
| `date_shift` | CPU: HMAC per row + redundant HKDF (F-3) | pandas date parsing |
| `fpe` | CPU: 8x HMAC per row (Feistel) | GIL-bound; threads help I/O latency not CPU |
| `faker` (pool path) | I/O: pool build if cache miss; HMAC at sample-time | Python loop in `_deterministic` |
| `shuffle` | Memory: numpy permutation | negligible |
| generation `reference` | CPU: `_apply_cardinality_bounds` O(n²) for large FK pools (F-4) | |
| distribution datetime | CPU: vectorized (fast); float quantization minor (F-8) | |

**What to profile:**
- `hash` / `date_shift` strategies: `python -m cProfile -s cumtime` on a 500K-row run.
  Look for `_derive.hkdf_extract`/`hkdf_expand` appearing ×500K — that's the F-3 bug.
- `_apply_cardinality_bounds`: run with `cProfile` on a 10K-row FK table with 500 parent
  values. Expect O(10K × 500) = 5M iterations; after the index-dict fix this drops to O(10K).
- `_match_source_cardinality`: `timeit` on a 100K-row MATCH_SOURCE_CARDINALITY column;
  compare `.iloc` loop vs `.tolist()` pre-materialization — expect 2-5x improvement.

**Algorithmic complexity summary:**

| Path | Current | After fix |
|------|---------|-----------|
| `_apply_cardinality_bounds` step 1+3 | O(n × |ref|) | O(n + |ref|) |
| `hash` strategy per-column HKDF | O(n) HKDF calls | O(1) HKDF + O(n) HMAC |
| `date_shift` per-column HKDF | O(n) HKDF calls | O(1) HKDF + O(n) HMAC |

| `sample_bundle` deterministic loop | O(n) with pandas overhead | O(n) pure Python |

## 5. What Is Good

- **HKDF implementation** (`determinism/_hkdf.py`): clean stdlib-only implementation
  pinned against RFC 5869 reference vectors. The explicit rejection of empty salts
  (QA-10 F12) and the single-responsibility split between `_hkdf.py` and `_derive.py`
  are textbook.

- **`DeriveContext`** (`determinism/_derive.py:110-170`): well-motivated pre-computation
  class with a clear docstring, tested identity claim, and backward-compatible coexistence
  with the scalar `derive()`. The `SEED_PROTOCOL_VERSION` changelog in the module header
  is exemplary: every bump has a date, a rationale, and a "no manifests to break"
  confirmation.

- **`_apply_null_probability` / `generate_column` null injection** (`synthesize.py:329-363`,
  `columns.py:239-275`): the V2 fix using `np.random.default_rng` + vectorized mask is
  correct and the dtype-promotion guard (Int64 before nullable assignment) closes a
  subtle pandas footgun.

- **`atomic_swap_db_providers`** (`internal/faker_setup.py:111-153`): the single-lock
  atomic swap over all three dictionaries (unregister + fn_map + val_map) is exactly
  right. The prior multi-step unregister-then-reregister was a TOCTOU window.

- **`_canonicalize_source` integer encoding** (`generation/pool/_canonicalize.py:44-56`):
  the shift from fixed 8-byte to length-prefixed minimal two's-complement (ASN.1 DER
  INTEGER pattern) correctly handles unbounded-magnitude Python ints and numpy integer
  scalars, and the table at the module header is a clear cross-sprint contract.

- **`FpeStrategyHandler._encrypt_values`** chunked threading: correctly caps workers at
  `os.cpu_count()` (avoids contention on GIL-bound pure-Python work) and preserves
  byte-identical output regardless of worker count — the determinism contract is met by
  construction, not by coordination.

- **`_generate_reference_column` typed errors** (`columns.py:912-926`): replacing silent
  sentinel strings (`"REF_TABLE_NOT_FOUND_0"`) with explicit `ValueError` is exactly
  right for an internal boundary.
