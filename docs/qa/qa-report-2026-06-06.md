**Scope:** `decoy-engine` execution pipeline, generation/pool, transforms, determinism/crypto.
**Companion:** `decoy-platform` report covers `api/jobs/db_io.py`, job runner, auth.
**Reviewer role:** Senior QA + performance engineer.

---

## 1. Summary

The engine's determinism envelope (HKDF-SHA256 + HMAC, S3/S5/S9) is correct and well-tested. The sprint cadence of named QA fixes (QA-1 through QA-10, S21 Q-series, Dennis C1) reflects a healthy correctness-first culture and the code shows it: no unseeded RNG leaks into the deterministic path, FK integrity is maintained through sorted pools and namespace binding, and the Feistel+HMAC FPE is a sound bijective cipher.

The most important open issue is **formula RNG isolation**: formula columns that use `randint`/`choice`/`random` are seeded from `(col_name, expr)` rather than from `job_seed`, so re-running with a different job seed does not change formula RNG output. This silently breaks the operator expectation that seed rotation produces fresh pseudonyms.

The second cluster of issues is **per-row `.iloc` overhead** in `_sampler.py` and `_categorical.py`: the S21 Q6 batch-materialization fix was applied to `_deterministic` but not to `_match_source_cardinality`, `sample_bundle`, or the categorical row loop, leaving performance gaps at scale.

## 2. Findings

### F1 — High | Correctness / Determinism
**Formula RNG is not job-seed-isolated.**

**Files:** `src/decoy_engine/transforms/formula.py:54`, `src/decoy_engine/execution/_strategies/_formula.py:37`

The formula seed is derived as:
```python
formula_seed = int(hashlib.sha256(f"{col_name}|{expr}".encode()).hexdigest()[:16], 16)
```
This is a function of `(col_name, expr)` only. `ctx.job_seed` is passed to `FormulaStrategyHandler.run` but never consumed — the formula strategy ignores it entirely.

**Impact:** Two masking jobs that share the same column name and formula expression but have different `job_seed` values will produce **byte-identical** output for any cell whose formula uses `randint`, `choice`, or `random`. Operators who rotate `job_seed` between production runs to produce distinct pseudonyms get no isolation for formula columns. This also means a formula-masked column cannot be joined across jobs using the namespace registry's same-namespace guarantee, since that guarantee assumes output is keyed on job_seed.

**Fix:** Feed `job_seed` into the formula seed derivation. The cleanest form is:

```python
# In FormulaStrategyHandler.run:
seed_material = ctx.job_seed + f"{column}|{cfg.get('formula','')}".encode()
formula_seed = int(hashlib.sha256(seed_material).hexdigest()[:16], 16)
rng = random.Random(formula_seed)
scope = make_mask_globals(rng)
df[column] = _V1_FORMULA.apply_with_scope(df[column], rule, scope)
```

This requires `FormulaStrategy.apply` to accept an external scope or `rng` so the strategy handler can inject the job-seeded instance. The V1 path (`_V1_FORMULA.apply`) already delegates to `make_mask_globals(rng)` — the only missing step is threading `ctx.job_seed` into the seed material before constructing `rng`. Until this is fixed, formulas using RNG are effectively seeded from a constant (their own text), not from the job.

**Verify:** Run the same column + formula with two different `job_seed` values; with the fix, outputs should differ. Property test: `∀ seed1 ≠ seed2, formula_rng_output(seed1) ≠ formula_rng_output(seed2)` for any formula containing `randint(0, 1000000)`.

### F2 — Medium | Performance
**`_match_source_cardinality` uses per-row `.iloc[i]` (S21 Q6 pattern not applied).**

**File:** `src/decoy_engine/generation/pool/_sampler.py:241-246`

```python
# Current code
is_null = source.isna()          # pd.Series — O(n) numpy call, good
for i in range(n):
    if is_null.iloc[i]:          # pandas scalar unbox per row — bad
        ...
    else:
        output.append(value_map[source.iloc[i]])  # pandas scalar unbox per row — bad
```

The `_deterministic` method 12 lines above (starting line 173) was fixed by S21 Q6 to use:
```python
src_values = source.tolist()
is_null_arr = source.isna().to_numpy()
```
The same fix was not applied to `_match_source_cardinality`.

**Impact:** At 100K rows, `.iloc[i]` adds ~50 µs/row of pandas scalar-unboxing overhead (verified by the S21 Q6 measurement). For MATCH_SOURCE_CARDINALITY on a 1M-row column with 50% nulls, this is ~25 s of pure pandas overhead vs. ~0.5 s for the materialized path.

**Fix:**
```python
def _match_source_cardinality(self, pool, n, source, rng, scale):
    ...
    src_values = source.tolist()              # one C-level list()
    is_null_arr = source.isna().to_numpy()   # one C-level mask
    output: list[Any] = []
    for i, val in enumerate(src_values):
        if is_null_arr[i]:
            output.append(pd.NA)
        else:
            output.append(value_map[val])
    return pd.Series(output)
```

### F4 — Medium | Performance
**`CategoricalStrategyHandler` deterministic path iterates over a pandas Series.**

**File:** `src/decoy_engine/execution/_strategies/_categorical.py:153, 167`

```python
# Current code — both uniform and weighted paths
for i, value in enumerate(source):   # pandas Series.__iter__: scalar-unboxing per row
    if na_mask[i]:   # na_mask is numpy, good
        ...
    idx = derive_index(..., _canonicalize_source(value), ...)
```

`na_mask` is correctly materialized to numpy (line 140). But `source` is still a pandas Series, and `enumerate(source)` pays the pandas iterator overhead (boxing/unboxing each scalar). The `_deterministic` sampler avoids this by calling `source.tolist()` first.

**Impact:** For a 1M-row column, iterating a pandas Series costs ~3-4× more than iterating a plain Python list. The `derive_index` HMAC cost dominates, but the iterator overhead is measurable (~0.3 s per million rows on a 2-vCPU runner — benchmarkable with `timeit` on `for v in series` vs `for v in series.tolist()`).

**Fix:**
```python
src_values = source.tolist()
for i, value in enumerate(src_values):
    if na_mask[i]:
        ...
```

### F5 — Medium | Performance
**`transforms/fpe.py:_encrypt` reconstructs `charset_set = set(charset)` on every call.**

**File:** `src/decoy_engine/transforms/fpe.py:215`

`_encrypt` is called once per non-null value from `apply()`. Line 215 creates a new `set(charset)` on each call. For a "digits" charset this is 10-element set construction 1M times; for "ALPHANUM" (62 chars) the cost is proportionally higher.

The `_CHARSET_INDEX` pre-computation at module load (lines 50-52) correctly amortizes the `char_to_idx` dict for `_fpe_pure`, but the `charset_set` used for membership testing in `_encrypt` missed the same optimization.

**Impact:** 1M set constructions + 1M GC events. At ~200 ns per `set()` call on CPython 3.10, this is ~0.2 s of pure overhead per 1M rows — non-trivial next to the Feistel cost (~8 HMAC-SHA256s per value ≈ ~4 µs ≈ 4 s/1M rows). Profile with `cProfile` on `apply()` to confirm.

**Fix:** Compute `charset_set` once in `apply()` and pass it to `_encrypt`:

```python
def apply(self, column, rule):
    ...
    charset_set = frozenset(charset)   # once per strategy invocation
    ...
    encrypted = [
        self._encrypt(s, key, charset, charset_set, tweak, ...)
        for s in non_na_str
    ]

def _encrypt(self, val, key, charset, charset_set, tweak, ...):
    if preserve_sep:
        for i, ch in enumerate(val):
            if ch in charset_set:   # O(1) lookup, same as before
                ...
    else:
        if not all(ch in charset_set for ch in val):
            ...
```

The same fix applies to `_strategies/_fpe.py` where `_V1_FPE._encrypt` is called inside the `encrypt_one` closure.

### F6 — Medium | Performance / Design
**`safe_eval` instantiates a new `EvalWithCompoundTypes` per formula evaluation (once per row).**

**File:** `src/decoy_engine/expressions.py:142-146`

```python
def safe_eval(expr, globals_, locals_):
    scope = {k: v for k, v in globals_.items() if k != "__builtins__"}  # dict copy per row
    scope.update(locals_)                                                 # per-row merge
    functions = {k: v for k, v in scope.items() if callable(v)}        # filter per row
    return EvalWithCompoundTypes(names=scope, functions=functions).eval(expr)  # object alloc per row
```

For a 100K-row formula column, this is 100K dict copies + 100K `EvalWithCompoundTypes` object constructions + 100K `eval` calls. The simpleeval instantiation alone adds measurable overhead on top of the expression evaluation.

**Impact:** Benchmarkable with `timeit("EvalWithCompoundTypes(names={...}).eval('value + 1')", ...)`. On CPython 3.10, instantiation is ~15 µs, making it ~1.5 s of pure overhead per 100K rows. This compounds with the pandas `apply` overhead.

**Fix:** Pre-build the evaluator once per column invocation and update `names["value"]` per row:

```python
def safe_eval_column(expr: str, globals_: dict, column: pd.Series) -> pd.Series:
    scope = {k: v for k, v in globals_.items() if k != "__builtins__"}
    functions = {k: v for k, v in scope.items() if callable(v)}
    evaluator = EvalWithCompoundTypes(names=scope, functions=functions)

    def apply_one(v):
        if pd.isna(v):
            return v
        evaluator.names["value"] = v
        return evaluator.eval(expr)

    return column.apply(apply_one)
```

This is a one-object-per-column pattern instead of one-object-per-row. The `names` dict mutation is safe because `FormulaStrategy.apply` is single-threaded (the `_FAKER_CALL_LOCK` pattern from `synthesize.py` is only needed for Faker; simpleeval has no shared state beyond the `names` dict).

### F7 — Low | Performance
**`_strategies/_fpe.py`: threading FPE over GIL-bound Feistel offers limited gain on pure-Python paths.**

**File:** `src/decoy_engine/execution/_strategies/_fpe.py:91-106`

The comment correctly documents that the Feistel cipher is "GIL-bound pure Python (only the stdlib-HMAC digest releases the GIL)." For a 2-vCPU runner, `workers = min(4, 2) = 2`; the GIL serializes the two threads on HMAC state manipulation, so actual parallelism is near-zero.

**Impact:** Thread creation + context-switching overhead (~5-50 µs per pool submission) partially offsets the wall-clock gain from GIL release during the hash digest. On a 4-vCPU machine the threading is mildly positive; on a 1-2 vCPU container it may be net-negative. Measurement: run `_encrypt_values` with `workers=1` vs. `workers=os.cpu_count()` on 100K values with `time.perf_counter_ns()`. If workers=1 is faster on 2 vCPU, cap `workers` at 1 for that configuration.

**Recommendation:** Add a `min_values_for_threading` threshold (e.g., 10 000) below which the serial path is always used. The current `len(values) < workers` guard handles the empty case but not the "threading overhead > gain" case for small columns.

### F8 — Nit | Reliability / Design
**`MASK_GLOBALS` retains module-level `_random.*` bindings as the default scope.**

**File:** `src/decoy_engine/expressions.py:99-101`

```python
MASK_GLOBALS: dict[str, Any] = {
    ...
    "randint": _random.randint,   # module-global RNG state
    "choice": _random.choice,
    "random": _random.random,
}
```

The QA-1 M21 fix correctly moved `FormulaStrategy` to use `make_mask_globals(rng)`. But `MASK_GLOBALS` itself still exposes module-level RNG bindings as a trap for future callers that pass `MASK_GLOBALS` directly instead of calling `make_mask_globals`. The comment acknowledges this but doesn't enforce it.

**Fix:** Remove `randint`/`choice`/`random` from `MASK_GLOBALS` entirely and require callers to use `make_mask_globals(rng)`. If any caller legitimately wants a non-isolated RNG, they can call `make_mask_globals(random.Random())`. This makes the unsafe path explicit rather than the default.

## 3. Performance Notes

### Bottleneck classification by strategy

| Strategy | Bottleneck | Key cost |
|---|---|---|
| hash | CPU | HMAC-SHA256 per row (~0.5 µs/row) |
| shuffle | CPU | numpy permutation (vectorized, fast) |
| categorical (uniform) | CPU + pandas overhead | HMAC per row + Series iterator (F4) |
| categorical (weighted) | CPU | HMAC + bisect per row |
| date_shift | CPU | HMAC per row + strptime (vectorized, good) |
| fpe | CPU | 8× HMAC per row + `charset_set` overhead (F5) |
| faker | CPU + I/O | Pool build is one-time; sampler is vectorized |
| formula | CPU | `safe_eval` + `EvalWithCompoundTypes` per row (F6) |
| composite (deterministic) | CPU + pandas overhead | `derive_index` + `.iloc` per row (F3) |
| MATCH_SOURCE_CARDINALITY | pandas overhead + dict lookup | `.iloc` per row (F2) |

### Big-O summary

All strategies are O(n) in both time and space. There are no super-linear paths. The constant factors differ:
- Vectorized paths (shuffle, faker REUSE, redact, truncate, bucketize): ~0.1-0.5 µs/row
- HMAC-keyed row loops (hash, categorical, date_shift, fpe): ~0.5-5 µs/row
- Formula with simpleeval: ~15-20 µs/row (dominated by EvalWithCompoundTypes instantiation, fixable by F6)

## 5. What's Good

**Determinism envelope is correct.** HKDF-SHA256 + HMAC-SHA256 (RFC 5869 + RFC 2104) with length-prefixed HMAC input, fixed non-empty salt, and per-column namespace isolation is a sound design. The `DeriveContext` pre-computation (per-column HKDF amortization) shows awareness of the setup cost. No unseeded RNG was found anywhere in the deterministic execution path.

**S21 Q6 materialization was the right call.** The `_deterministic` sampler's `source.tolist()` + `source.isna().to_numpy()` pattern correctly identified that pandas scalar-unboxing cost was exceeding the HMAC cost. The fix is clean and the docstring explains why. F2/F3/F4 are follow-on applications of the same pattern.

**Formula sandbox (SEC.1 / C1) is well-reasoned.** The migration from `eval(expr, {"__builtins__": {}}, ...)` to `simpleeval.EvalWithCompoundTypes` is the right call. The documented rationale (dunder traversal, `.format` escape, module reference block) shows the threat model was understood rather than hand-waved.

**FK integrity under sorted pools.** `sorted(raw_pool)` + `dropna()` in `_generate_reference_column` (QA walks F1) is load-bearing: without it, FK pool assignment is order-dependent on DB row fetch order. The sort makes the mapping a pure function of the data, which is the right invariant.

**Error handling is explicit and typed throughout.** `StrategyError`, `ExecutionError`, `GenerationError` all carry `code` fields; `KeyResolutionError` raises instead of degrading. The "fail loud at boundaries" principle is consistently applied.

**db_io.py SQL injection defense is layered correctly.** The three-layer defense (character allowlist + keyword denylist + ORDER BY regex) is defense-in-depth appropriate for an admin-reachable path. The QA-5 L1 whitespace normalization fix (collapsing `\s+` before keyword match) catches the `INTO\tOUTFILE` bypass vector. The mandatory ORDER BY for DB-sourced mask jobs (QA-5 H2 raise instead of warn) correctly enforces the determinism contract at the boundary.
