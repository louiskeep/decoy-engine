# QA Review — `decoy-engine`

## 1. Summary

The determinism layer (`determinism/`) is well-engineered: RFC 5869 HKDF + HMAC-SHA256 with length-prefixed canonical inputs, a documented protocol version byte, and sound injection design. The critical problems sit in two other areas. First, `DBHandler.load_data()` issues queries without ORDER BY, making every database-backed run nondeterministic at the row level -- this silently breaks the reproducibility guarantee the entire system is built on. Second, the hot path for deterministic pool sampling (`PoolSampler._deterministic`) and FK key resolution (`_parent_map`, `_resolve_fk_node`) use Python-level `for i in range(n): ... .iloc[i]` loops that degrade to hundreds of seconds at 1M+ rows.

## 2. Findings

### #1 -- Critical -- Correctness / Determinism

**`connectors/database.py:load_data()` -- no ORDER BY; row ordering is nondeterministic**

```python
query = f'SELECT * FROM "{table}"'          # no ORDER BY
query = f'SELECT * FROM "{table}" WHERE {where}'
```

A database is free to return rows in any order across runs. The masking engine applies rules positionally (row *i* in the input maps to row *i* in the output). Two runs against the same table with the same seed will produce byte-identical per-value output but may write those values to **different rows**, so the "same seed -> same output" contract fails for any downstream consumer that reads row-by-position.

For chunked masking the problem compounds: chunk boundaries shift, so referential integrity windows (FK parent-before-child) can fall in different chunks across runs.

**Fix:**

```python
def load_data(self) -> pd.DataFrame:
    engine = self._get_in_engine()
    table = self.input_config["table"]
    order_col = self.input_config.get("order_by")          # required; fail loud if absent
    where = self.input_config.get("where")
    if not order_col:
        raise ValueError(
            f"DB connector for table '{table}' requires an 'order_by' key "
            "in input_config for deterministic row ordering."
        )
    parts = [f'SELECT * FROM "{table}"']
    if where:
        parts.append(f"WHERE {where}")          # see also finding #4
    parts.append(f'ORDER BY {order_col}')
    with engine.connect() as conn:              # see also finding #2
        return pd.read_sql(text(" ".join(parts)), conn)
```

**Verify:** run the same job twice, `diff` the outputs; they must be byte-identical.

### #2 -- Critical -- Correctness

**`connectors/database.py:close()` -- double-dispose when `_out_engine` falls back to `_in_engine`**

```python
def close(self) -> None:
    if self._in_engine is not None:
        self._in_engine.dispose()
        self._in_engine = None          # <- now None

    if self._out_engine is not None and self._out_engine is not self._in_engine:
        # ^ self._in_engine is ALREADY None here, so `is not None` is always True
        # even when _out_engine was sharing the same object
        self._out_engine.dispose()      # double-dispose on the same pool
```

When no separate `connector_dsn` is configured, `_get_out_engine()` returns `self._get_in_engine()` -- both attributes point to the same object. The close block disposes it once (fine), nulls `_in_engine`, then disposes `_out_engine` again via a pool that is already closed. SQLAlchemy raises on second `dispose()` or silently corrupts the pool depending on version.

**Fix:**

```python
def close(self) -> None:
    engines = {e for e in (self._in_engine, self._out_engine) if e is not None}
    for eng in engines:
        eng.dispose()
    self._in_engine = None
    self._out_engine = None
```

### #3 -- High -- Correctness

**`connectors/database.py:load_data()` -- connection leak via bare `engine.connect()`**

```python
return pd.read_sql(text(query), con=engine.connect())
```

`engine.connect()` creates a `Connection` that is never closed or returned to the pool. Under SQLAlchemy 2.x the connection stays checked-out until GC finalizes it (non-deterministic). Under connection-pool exhaustion this hangs the process.

**Fix:**
```python
with engine.connect() as conn:
    return pd.read_sql(text(query), conn)
```

### #4 -- High -- Correctness / Security

**`connectors/database.py:load_data()` -- raw WHERE clause is SQL injection**

```python
query = f'SELECT * FROM "{table}" WHERE {where}'
```

`where` is interpolated verbatim. Even if "config is trusted," configs are often sourced from files edited by operators, web UIs, or CI pipelines. A value like `1=1; DROP TABLE users; --` executes. The table name is double-quoted (good) but `where` is not.

Where structured filtering is needed, use parameterized fragments. Where free-form SQL is genuinely needed (admin feature), make that an explicit, logged, `allow_raw_sql: true` opt-in and wrap it in a guard:

```python
if self.input_config.get("allow_raw_sql"):
    # log a WARNING that raw SQL is active
    parts.append(f"WHERE {where}")
else:
    raise ValueError("use 'filter_column'/'filter_value' keys for parameterized filtering")
```

### #5 -- High -- Correctness

**`masker/masker.py:mask()` -- file-existence check always fails for DB connectors**

```python
input_path = self.config["input"].get("path", "")
...
if not os.path.exists(input_path):
    raise FileNotFoundError(error_msg)          # <- always fires when path=""

file_size_gb = os.path.getsize(input_path)      # would also fail
```

`os.path.exists("")` returns `False`. Any DB-backed job reaches `Masker.mask()` and immediately raises `FileNotFoundError: Input file not found:` before any SQL is issued. The large-file threshold check and the chunked / standard branch are also file-size driven and meaningless for DB sources.

**Fix:** gate on connector type before path checks:

```python
connector_type = self.config["input"].get("type", "csv")
if connector_type in ("db", "database"):
    self._process_db_source(table_name)
    return
if not input_path:
    raise ValueError(f"input.path is required for connector type '{connector_type}'")
if not os.path.exists(input_path):
    ...
```

### #6 -- High -- Performance

**`generation/pool/_sampler.py:_deterministic()` -- O(n) Python loop dominates hot path**

```python
for i in range(n):
    if is_null.iloc[i]:                         # pandas scalar unbox per row
        output.append(pd.NA)
        continue
    canonical = _canonicalize_source(source.iloc[i])   # Python call per row
    idx = derive_index(seed=seed, namespace=namespace,
                       source=canonical, pool_size=pool.size)
    output.append(pool.values[idx])
return pd.Series(output)
```

**Bottleneck classification: CPU-bound Python.** `derive_index` calls HKDF-SHA256 + HMAC-SHA256 per row -- that's irreducible at ~200K-500K ops/s per core. But the surrounding Python overhead (`.iloc[i]` unboxing, `append`, `pd.NA` branching) roughly doubles total wall time. At 1M rows this is ~5-15 s just for this loop on a single core; at 10M rows it becomes a minutes-scale bottleneck.

**Recommended improvement:** batch-convert the source to a Python list once, eliminate per-row pandas overhead:

```python
non_null_mask = ~source.isna()
src_list = source.tolist()          # one C-level materialise
output = [pd.NA] * n
for i, val in enumerate(src_list):
    if non_null_mask.iloc[i]:
        canonical = _canonicalize_source(val)
        output[i] = pool.values[derive_index(seed, namespace, canonical, pool.size)]
return pd.Series(output)
```

`source.tolist()` is ~10x faster per element than repeated `.iloc[i]`. For further gains (S13 gate), the HMAC calls could be parallelized in a `ThreadPoolExecutor` since each row is independent (no shared mutable state in `derive_index`).

**Verify:** `python -m timeit` with 100K-row Series input before and after; target >= 5x speedup.

### #7 -- High -- Performance

**`execution/_pandas_adapter.py:_parent_map()` and `_resolve_fk_node()` -- O(n) Python loops on large FK tables**

`_parent_map` (building source -> masked key dict):
```python
for i in range(n):
    raw = [s.iloc[i] for s in src_series]       # k x .iloc per row
    ...
    out[src_t] = tuple(s.iloc[i] for s in masked_series)
```

`_resolve_fk_node` (building child_keys):
```python
for i in range(n):
    if any(bool(na.iloc[i]) for na in col_na):  # per-row pandas overhead
        ...
    else:
        child_keys.append(tuple(_fk_key_value(v.iloc[i]) for v in col_vals))
```

For a parent table with 1M rows and a composite FK with 3 columns, `_parent_map` issues ~3M `.iloc[i]` calls. This is the main FK-resolution bottleneck.

**Recommended improvement for `_parent_map`** (eliminates Python loop entirely):

```python
src_df = pd.concat(src_series, axis=1)
masked_df = pd.concat(masked_series, axis=1)
null_rows = src_df.isna().any(axis=1)
valid = src_df[~null_rows]
src_tuples = [tuple(_fk_key_value(x) for x in row) for row in valid.itertuples(index=False)]
masked_tuples = [tuple(row) for row in masked_df[~null_rows].itertuples(index=False)]
out = dict(zip(src_tuples, masked_tuples))
```

`itertuples` is ~5-10x faster than repeated `.iloc[i]` for multi-column frame access.

### #8 -- Medium -- Performance

**`transforms/faker_based.py:_apply_value_seeded()` -- `column.apply()` is a Python-per-row loop; inconsistent with `hash.py` optimization**

```python
return column.apply(fake_for)
```

`Series.apply` with a Python callable is O(n) with per-element boxing overhead -- the exact anti-pattern the hash strategy comment explicitly documents avoiding. At 1M rows, Faker itself is the main cost (~5-50 us per call depending on `faker_type`), but removing `apply` in favour of the same list-comprehension pattern used in `hash.py` removes one unnecessary boxing layer and keeps the codebases consistent:

```python
values = column.tolist()
result_vals = [fake_for(v) for v in values]
result = pd.Series(result_vals, index=column.index, dtype=object)
return result
```

### #9 -- Medium -- Correctness / Security

**`transforms/hash.py` and `transforms/date_shift.py` -- both `_column_key()` call `derive_key("mask")` with identical label, sharing a derived key across strategy types**

```python
# HashStrategy._column_key:
return self.derive_key("mask")

# DateShiftStrategy._column_key (same):
return self.derive_key("mask")
```

Two different masking strategies operating in the same pipeline derive the same subkey. If an attacker obtains masked outputs from both strategies applied to correlated source values (e.g., hash of an email and date_shift of a birthdate that share source structure), the shared key makes certain correlation attacks easier. More practically, if a bug is ever introduced where a hash is accidentally compared against a date-shift output, the equality check could yield false matches.

**Fix:** include the strategy name in the derivation info:

```python
# HashStrategy:
return self.derive_key(f"mask:hash")

# DateShiftStrategy:
return self.derive_key(f"mask:date_shift")
```

### #10 -- Medium -- Security / Reliability

**`masker/masker.py:mask()` and `connectors/csv_connector.py` -- exception strings may contain PII**

```python
# masker.py
self.logger.error(f"Error during masking process: {e!s}")

# csv_connector.py
raise ValueError(f"Failed to load CSV file. Ensure the file has headers: {e}")
raise ValueError(f"Failed to load CSV file: {e}")
```

Exception messages from pandas often include cell values (e.g. `"ValueError: time data '555-John-Smith' does not match format '%Y-%m-%d'"`). These get logged or surface in API error responses, leaking original sensitive values into logs and error payloads -- precisely what masking is meant to prevent.

**Fix:** log exception type and a sanitized message only; never `str(e)` for data-parsing errors:

```python
self.logger.error(
    f"Error during masking process: {type(e).__name__} "
    f"(column={rule.get('column', 'unknown')!r})"
)
```

### #11 -- Medium -- Design

**`context.py:_hkdf_sha256()` -- second, divergent HKDF implementation; uses zero salt**

`context.py` has its own one-round HKDF:
```python
salt = b"\x00" * 32
prk = hmac.new(salt, master, hashlib.sha256).digest()
okm = hmac.new(prk, info.encode("utf-8") + b"\x01", hashlib.sha256).digest()
return okm[:length]
```

`determinism/_hkdf.py` has a full RFC 5869 implementation. These are two different derivations. A future author may assume they're equivalent when they're not (different salts, different concatenation). The `length > 32` guard in `context.py` prevents multi-round use and would silently truncate at the wrong boundary if someone passed `length=33`.

**Fix:** import and use `hkdf_sha256` from `determinism/_hkdf.py` in `context.py`. Define a module-level constant for the zero-salt if it is intentional:

```python
from decoy_engine.determinism._hkdf import hkdf_sha256

def _make_key_resolver_key(master: bytes, info: str, length: int = 32) -> bytes:
    return hkdf_sha256(ikm=master, salt=b"\x00" * 32, info=info.encode("utf-8"), length=length)
```

### #12 -- Low -- Reliability

**`connectors/database.py:save_data()` -- `to_sql` without explicit transaction; partial writes are not rolled back**

```python
df.to_sql(table, con=engine, if_exists=if_exists, index=False)
```

For large DataFrames, `to_sql` emits many INSERT batches. If one fails mid-write with `if_exists='replace'`, the table has already been dropped and only partially refilled. No rollback occurs.

**Fix:**
```python
with engine.begin() as conn:     # auto-commits on success, auto-rollbacks on exception
    df.to_sql(table, con=conn, if_exists=if_exists, index=False,
              chunksize=10_000,   # also improves throughput vs default row-by-row
              method="multi")
```

### #13 -- Low -- Correctness

**`transforms/shuffle.py` -- `dtype=column.dtype` then `shuffled[na_mask] = None` coerces integer columns to float**

```python
shuffled = pd.Series(index=column.index, dtype=column.dtype)
shuffled[na_mask] = None   # <- promotes int64 -> float64 in non-nullable integer columns
```

Setting `None` on a non-nullable int64 Series triggers silent dtype promotion to float64. The output column type changes from the input type, which breaks downstream dtype contracts.

**Fix:** use `pd.array` with nullable types or explicitly handle the null promotion:

```python
result_values = column.to_numpy(dtype=object, na_value=pd.NA, copy=True)
result_values[~na_mask] = shuffled_values
return pd.Series(result_values, index=column.index)
```

### #14 -- Nit -- Design

**`masker/masker.py` -- default seed of `42` is well-known; no warning when user omits it**

```python
seed = self.config.get("global_settings", {}).get("seed", 42)
```

Using a publicly known default means a user who forgets to set a seed gets reproducible-but-guessable output. At minimum, log a `WARNING` when falling back to the default:

```python
seed = self.config["global_settings"].get("seed")
if seed is None:
    self.logger.warning(
        "No 'seed' set in global_settings. Using default seed=42; "
        "set an explicit seed for production runs."
    )
    seed = 42
```

## 3. Performance Notes

| Bottleneck | Location | Estimated impact |
|---|---|---|
| `_deterministic` Python loop | `pool/_sampler.py:~100-115` | ~10 s at 1M rows (CPU-bound Python); HMAC is irreducible, loop overhead is not |
| `_parent_map` Python loop | `_pandas_adapter.py:~195-206` | ~5-10 s at 1M-row parent table |
| `_resolve_fk_node` Python loop | `_pandas_adapter.py:~166-177` | ~3-8 s at 1M-row child table, k FK columns |
| `FakerStrategy.apply` | `transforms/faker_based.py:~73` | Faker itself is ~5-50 us/row; `apply()` adds 1-3 us overhead per row at 1M = 1-3 s extra |
| `DBHandler.load_data` | `connectors/database.py:~49` | Connection leak forces GC-based cleanup; under pool exhaustion -> hang |

**Profiling recipe:** `python -m scalene --cpu --memory --outfile profile.html -m pytest tests/benchmark/` -- Scalene annotates per-line CPU and memory. For the HMAC-heavy deterministic path specifically, `py-spy record -o flame.svg -- python your_script.py` gives a flamegraph to confirm `.iloc` vs. `hmac.new` share of time.

**Memory:** The masker materializes the entire DataFrame in memory for the standard path. For a 1 GB CSV with 20 columns of object dtype, peak memory is typically 3-5x the file size (CSV string parsing overhead + pandas object array). The large-file threshold at 1 GB is a reasonable guard, but the threshold check uses `os.path.getsize(input_path)` which measures compressed or raw file size, not in-memory size -- on gzipped CSVs the threshold fires too late.

## 5. What's Good

The **determinism envelope** in `determinism/_derive.py` and `_hkdf.py` is the best-written section: the HMAC input is length-prefixed (injective), the HKDF salt binds the primitive type, the version byte is mixed into HMAC input (not the salt), and the `_canonicalize_source` dispatch handles the numpy-scalar edge case (integers via `numbers.Integral` ABC) that trips most naive implementations. The decision to hard-error on float and tz-naive datetime rather than silently baking fragile rules is the right call. The pool cache in `_pool_adapter.py` and the `identity_for` / build separation (S5 F1 fix) correctly avoids building pools on cache hits.

The **Kahn topological sort** in `_runner.py` is deterministic (sorted tie-break), handles the R17 composite-before-FK-child edge, and explicitly errors on cycles rather than silently looping. The FK snapshot-before-mask pattern in `_pandas_adapter.py` correctly ensures child resolution uses pre-mask parent keys even when parent and child are in the same batch.

The **`hash.py` optimization comment** (vectorized null mask + list comprehension vs. `Series.apply`) documents exactly why the optimization exists -- the kind of comment that should be copied as a pattern to `faker_based.py` and the sampler loops.
